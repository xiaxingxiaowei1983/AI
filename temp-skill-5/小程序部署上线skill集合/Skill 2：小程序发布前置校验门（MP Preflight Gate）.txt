# Skill 2：小程序发布前置校验门（MP Preflight Gate）

## 目标
在你“上传体验版/提审/上线”之前，用最短路径做一次**一票否决**预检：
- 通过：进入下一步（微信后台配置校验/适配验收/提审）
- 不通过：立刻停止，按固定定位路径处理，不进入长对话排查

> 小程序最常见的崩溃点不是代码，而是：HTTPS/WSS、白名单、权限隐私、包体积、入口口径漂移。本 skill 专门提前挡住这些“必失败项”。

## 何时调用（触发条件）
- 每次准备上传小程序体验版之前（staging/prod 都要）
- 每次改了域名、证书、WS 路径、权限/隐私文案之后
- 出现这些现象时：
  - “H5 能用，小程序不通”
  - “接口请求失败，但看不懂原因”
  - “WS 连不上/一直断开”
  - “上传体验版失败/提示包过大/提审被拒”

- 目标环境：`staging` / `prod`
- MP Freeze Spec 版本号：v__
- AppID：
- API Base URL（必须 HTTPS）：
- WSS 域名 + WS 路径（若启用实时）：`wss://____` + `/ws` <mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 是否启用实时：是/否
- 是否涉及录音/音频：是/否
- 隐私政策链接：
- 是否启用分包：是/否（不确定就填“不确定”）
- 后端真实端口（内部）：____（例 3006）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>

## 前置约束（硬规则）
- 任何一条 Gate 失败：本次发布流程停止
- 本 skill 只输出：PASS/FAIL + 证据 + “前三定位路径”
- 不做“猜测式长排查”，不展开到下一个 skill

## 预检清单（8 条硬检查）
每条必须输出：✅/❌ + 证据 + 下一步

### Gate 1：入口口径一致（跟 MP Freeze Spec 完全一致）
**检查目标**：防止“写了 A、实际用 B”的口径漂移  
- 证据要求（任选其一）：
  - 已生成 MP Freeze Spec 结果单，且本次发布输入与之完全一致
  - 小程序端配置/常量中可定位到 API Base、WS 路径，且与 Spec 一致
- FAIL 的前三定位路径：
  1) 是否同时存在多套环境配置（dev/staging/prod 混用）
  2) 是否有硬编码旧域名/旧路径残留
  3) 是否构建产物仍是旧配置（未重新打包/未刷新）

### Gate 2：HTTPS/WSS 可用（小程序强制）
**检查目标**：杜绝 http/ws 或证书问题导致“必失败”  
- 证据要求：
  - API Base 是 `https://`
  - 实时域名是 `wss://`（若启用）
  - 证书有效且域名匹配（至少能被手机正常访问，不弹证书风险）
- FAIL 的前三定位路径：
  1) 证书是否与域名匹配、是否完整链
  2) 是否存在跳转到 http（301/302）
  3) WS 是否实际走了 ws 而不是 wss

### Gate 3：微信后台白名单“准备就绪”（至少确认清单，不要求已填完）
**检查目标**：防止你上传后才发现“域名没法用/没备案/没白名单”  
- 证据要求：
  - 已列出准备填写的合法域名清单：request/socket/upload/download/业务域名（如有）
  - 域名可公开访问且是 HTTPS
- FAIL 的前三定位路径：
  1) 域名是否可访问（DNS/解析是否正确）
  2) 域名是否备案导致微信后台无法配置（大陆场景常见）
  3) 是否漏了 socket 合法域名（实时必需）

### Gate 4：后端可用性（接口至少一条轻量验证）
**检查目标**：避免“其实后端挂了，你还在折腾小程序”  
- 证据要求：
  - 轻量接口可用（200 或业务成功码）
  - 若采用反代：对外入口可用（https 域名访问成功）
- FAIL 的前三定位路径：
  1) 后端服务是否 running、端口是否监听
  2) Nginx/网关是否反代到正确上游端口
  3) env 是否读错（prod 配置缺失/错误）

### Gate 5：实时通道“基本可达”（若启用实时）
**检查目标**：避免提审前完全没跑通实时  
- 证据要求（任选其一）：
  - 真机能建立 WSS 连接并收到一条服务器消息/心跳
  - 后端能看到一次连接日志（connect/握手）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- FAIL 的前三定位路径：
  1) WS 路径是否一致（通常 `/ws`）
  2) Nginx/网关是否支持 Upgrade
  3) 协议是否一致（https 页面必须 wss）

### Gate 6：录音权限与隐私“最低可提审状态”（若涉及录音）
**检查目标**：避免审核直接打回  
- 证据要求：
  - 隐私政策链接可访问
  - 已确认需要的权限：录音（以及其他）
  - 已准备“用户数据收集说明”（一句话即可，但必须有）
- FAIL 的前三定位路径：
  1) 隐私政策链接是否失效/打不开
  2) 功能描述与权限不一致（说不采集但实际录音）
  3) 未做拒绝授权的兜底提示

### Gate 7：包体积/分包“不会卡在上传”
**检查目标**：避免你最后一步卡在“包太大/上传失败”  
- 证据要求（任选其一）：
  - 已确认使用分包（或当前包体积明显很小且无大资源）
  - 已清理不必要的资源（大音频/大图片/调试文件）
- FAIL 的前三定位路径：
  1) 静态资源是否过大（音频/图片）
  2) 是否需要开启分包与按需加载
  3) 是否把不该进包的文件打进去了

### Gate 8：真机基础冒烟（至少 3 条）
**检查目标**：避免“模拟器能用，真机不行”  
- 证据要求（最少 3 条）：
  - 能打开首页
  - 能请求一条轻量 API 成功
  - 若涉及录音：能弹出授权并完成一次录音开始/停止
- FAIL 的前三定位路径：
  1) 真机网络/权限问题（不是代码）
  2) 白名单/HTTPS/WSS 问题（平台拦截）
  3) 版本/配置没更新（还是旧包）

## 输出格式（固定，直接复制到发布记录）
- MP Preflight 结论：PASS / FAIL
- 环境：staging/prod
- Spec 版本：v__
- Gate1 入口一致：✅/❌（证据：___）
- Gate2 HTTPS/WSS：✅/❌（证据：___）
- Gate3 白名单准备：✅/❌（证据：___）
- Gate4 后端可用：✅/❌（证据：___）
- Gate5 实时可达：✅/❌（证据：___）
- Gate6 隐私权限：✅/❌（证据：___）
- Gate7 包体积：✅/❌（证据：___）
- Gate8 真机冒烟：✅/❌（证据：___）
- 若 FAIL：本次停止；前三定位路径：P1___ / P2___ / P3___

## 完成标准（本 skill 完成的判定）
- 你得到明确 PASS/FAIL
- 每个 Gate 都有“证据”（不是主观判断）
- FAIL 时流程已停止，并给出前三定位路径（不进入下一 skill）
