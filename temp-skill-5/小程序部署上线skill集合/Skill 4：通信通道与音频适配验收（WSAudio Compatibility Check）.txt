# Skill 4：通信通道与音频适配验收（WS/Audio Compatibility Check）

## 目标
把你们这类“实时 + 语音”的核心链路一次性验收清楚，避免最折磨人的情况：
- API 能用，但实时不通
- 实时能连，但音频上行后端不认
- H5 能跑，小程序不跑（因为通道与格式不一样）

你们项目的关键差异口径（冻结）：
- H5：Socket.IO（`/socket.io`）
- 小程序：原生 WebSocket（`/ws`）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 音频上行：小程序常见 `base64 + format`，后端需兼容解析 <mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>

## 何时调用（触发条件）
- 小程序开始联调实时/语音功能时（第一次必跑）
- 每次改 WS 路径、Nginx/网关反代、证书、音频格式字段时
- 出现这些现象时：
  - “API OK，但 WS 连接不上/很快断开”
  - “录音能录，但上传后无响应/解析失败”
  - “H5 好的，小程序坏的”

## 输入（固定表单）
- 目标环境：staging / prod
- API Base URL（HTTPS）：
- 是否启用实时：是/否
- WSS 入口（域名 + 路径）：`wss://____` + `/ws`（你们常用）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 后端真实端口（内部）：____（例 3006）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 是否涉及音频：是/否
- 小程序音频上行格式（冻结）：base64 + format（或你们实际值）<mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>
- format 允许值（冻结）：____（如 pcm/wav/mp3）

## 前置约束（硬规则）
- 小程序侧必须使用 **WSS**（https 页面下的安全通道）；不能用 ws
- 任一关键项 FAIL：结论 FAIL（停止进入提审/上线）
- 证据必须来自“真机或体验版”，不以模拟器替代

## 验收清单（按顺序执行）
### Check 1：WS 握手是否成功（连接建立）
- 通过标准：
  - 小程序端连接回调成功（连接状态为 open/connected）
  - 后端能看到一次“连接建立”的证据（日志关键字/在线连接数变化）
- 证据：
  - 小程序端调试面板/日志截图（连接成功）
  - 后端日志关键字或连接统计（任选一个）

FAIL 的前三定位路径：
1) 微信后台 socket 合法域名是否配置（先排平台侧）
2) 协议是否为 wss，证书是否正确
3) Nginx/网关是否支持 Upgrade，WS 路径是否一致（通常 `/ws`）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>

### Check 2：WS 基础收发是否成功（Ping/Pong 或一问一答）
- 通过标准：
  - 小程序发送一条最小消息（例如 `ping` 或 `{type:'hello'}`）
  - 服务端返回一条确认消息（例如 `pong` 或 `{type:'ack'}`）
- 证据：
  - 小程序端发送/接收日志截图（或服务端日志）

FAIL 的前三定位路径：
1) 服务端 WS 路由是否正确处理消息类型（消息格式不对会被丢弃）
2) 是否有心跳/超时导致断连（代理超时/服务端策略）
3) 是否连接到了错误环境（staging/prod 混用）

### Check 3：音频权限与录音“最小闭环”（若涉及音频）
- 通过标准：
  - 首次授权弹窗正常
  - 能开始录音、能停止录音
  - 拒绝授权时有提示与降级方案（至少不崩）
- 证据：
  - 真机录音开始/停止截图或日志
  - 拒绝授权提示截图（可选但推荐）

FAIL 的前三定位路径：
1) 小程序权限声明/隐私弹窗是否齐（审核/运行都可能卡）
2) 录音 API 是否被机型限制（换一台机型验证）
3) 是否因网络失败误判为录音失败

### Check 4：音频上行格式是否符合约定（若涉及音频）
- 通过标准：
  - 小程序发送音频消息字段符合约定（例如 `{type:'audioData', audio: base64, format}`）
  - base64 解码后大小合理（不是 0 或异常小）
- 证据：
  - 小程序端发送的字段截图（隐藏敏感信息即可）
  - 服务端日志/返回中能确认“收到音频长度/format”

FAIL 的前三定位路径：
1) format 字段是否缺失/拼写不一致（最常见）
2) base64 是否带了 dataURL 前缀导致解析失败
3) 分片/合并策略是否一致（一次发太大可能被限制）

### Check 5：后端解析与业务结果是否成功（若涉及音频）
- 通过标准（最少满足其一）：
  - 后端成功把音频转成 Buffer 并进入后续处理流程（有日志证据）
  - 或后端返回了明确业务结果（如识别文本/处理成功码）
- 证据：
  - 后端日志关键字（“audio received/decoded/format=..”等）
  - 或接口/WS 返回结果截图

FAIL 的前三定位路径：
1) 后端是否只兼容 H5 的 ArrayBuffer，没有兼容小程序 base64 <mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>
2) 音频 format 是否在后端支持列表内
3) 上传大小/超时限制（网关/Nginx/后端 body 限制）

### Check 6：断线重连策略（强烈建议验）
- 通过标准：
  - 断网/切后台后能重连（或至少有明确提示）
  - 不出现“假连接”（显示已连但实际收发失败）
- 证据：
  - 重连成功日志/截图

FAIL 的前三定位路径：
1) 心跳策略不匹配（服务端/客户端）
2) 代理超时（网关/Nginx）
3) 小程序后台限制（需要产品设计降级）

## 验收点（可量化）
- A1：WSS 连接建立成功（有证据）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- A2：最小收发成功（ping/pong 或 ack）
- A3：若涉及音频：录音权限/开始/停止闭环 OK
- A4：若涉及音频：base64+format 上行字段符合约定 <mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>
- A5：后端解析成功并产生业务结果（或进入处理流程）
- A6：断线重连可用或有明确降级提示

## 输出格式（固定，直接复制到发布记录）
- WS/Audio Compatibility：PASS / FAIL
- 环境：staging/prod
- WSS 入口：wss://____ + /ws
- Check1 连接建立：✅/❌（证据：___）
- Check2 基础收发：✅/❌（证据：___）
- Check3 录音闭环：✅/❌（证据：___）
- Check4 上行格式：✅/❌（证据：___）
- Check5 后端解析：✅/❌（证据：___）
- Check6 重连策略：✅/❌（证据：___）
- 若 FAIL：本次停止；前三定位路径：P1___ / P2___ / P3___

## 完成标准（本 skill 完成的判定）
- 你拿到 PASS/FAIL
- 每个检查项都有证据（截图/日志关键字/返回结果）
- FAIL 时停止进入提审/上线，先按定位路径修正再重跑
