# BUG修复执行与验收

## 任务目标
- 本Skill用于：执行BUG修复方案并进行验收，从方案执行到问题汇总的完整闭环
- 能力包含：进度跟踪、异常处理指导、验收标准制定、验收检查、总结复盘
- 触发条件：已完成执行规划，准备执行修复方案时
- 前置条件：已完成执行规划，有详细的任务清单和计划

## 前置准备
- 确保已完成执行规划，有详细的执行计划
- 准备好测试环境、部署环境
- 验收标准参考见 [bug-fix-debugger/references/verification-checklist.md](../bug-fix-debugger/references/verification-checklist.md)

## 操作步骤

### 步骤13：执行方案

#### 13.1 按计划执行修复
- **执行任务清单**：
  - 按照任务清单逐步执行
  - 完成每个任务后更新状态
  - 记录每个任务的完成时间和输出物

- **遵循依赖关系**：
  - 确保前置任务完成后再执行后续任务
  - 遇到阻塞及时沟通协调

- **记录关键决策**：
  - 记录执行过程中的关键决策
  - 记录计划变更的原因
  - 记录临时方案的选择理由

#### 13.2 遇到异常时启动应对策略
- **识别异常**：
  - 监控任务执行状态
  - 及时发现异常情况

- **启动应对**：
  - 按照异常应对计划执行
  - 评估异常影响
  - 决定是否需要调整计划

- **记录异常**：
  - 记录异常发生的时间
  - 记录异常的原因分析
  - 记录应对措施和结果

**智能体能力**：进度跟踪、异常处理指导

### 步骤14：验收标准

#### 14.1 制定验收标准
根据修复方案制定明确的验收标准。

**通用验收标准**：

**1. 功能验证**（MUST）
- [ ] **问题彻底解决**
  - 原问题不再复现
  - 在相同条件下测试，问题不再发生
  - 在不同边界条件下测试，问题仍然不发生

- [ ] **期望行为正常**
  - 功能按预期正常工作
  - 用户交互体验正常
  - 边界条件处理正确

- [ ] **多场景验证**
  - 正常场景：功能正常
  - 异常场景：有适当的错误处理
  - 边界场景：边界值处理正确
  - 并发场景：多线程/多用户情况下正常

**2. 回归测试**（MUST）
- [ ] **关联功能不受影响**
  - 直接关联的功能正常
  - 间接关联的功能正常
  - 跨模块调用正常

- [ ] **核心功能验证**
  - 用户登录/注册
  - 核心业务流程
  - 数据读写操作
  - 权限控制

- [ ] **不引入新BUG**
  - 修复代码路径无新增BUG
  - 修改未影响其他代码路径
  - 日志中无新增错误

**3. 性能评估**（SHOULD）
- [ ] **响应时间**
  - 修复后响应时间无明显增加（增长<10%）
  - 关键接口响应时间在可接受范围内

- [ ] **资源占用**
  - CPU占用无明显增加
  - 内存占用无明显增加
  - 磁盘I/O无明显增加

- [ ] **并发性能**
  - 高并发场景下系统稳定
  - 无死锁或竞态条件
  - 事务处理正常

**4. 数据完整性**（MUST）
- [ ] **数据一致性**
  - 修改数据正确保存
  - 关联数据同步更新
  - 数据库事务正确提交/回滚

- [ ] **数据准确性**
  - 计算结果正确
  - 数据精度符合要求
  - 数据格式正确

- [ ] **数据安全**
  - 敏感数据加密
  - 数据脱敏处理
  - 权限控制有效

**5. 代码质量**（SHOULD）
- [ ] **代码规范**
  - 符合项目编码规范
  - 命名清晰规范
  - 代码结构合理

- [ ] **代码可读性**
  - 关键逻辑有注释
  - 变量命名准确
  - 函数职责单一

- [ ] **代码复杂度**
  - 圈复杂度在合理范围
  - 避免深层嵌套
  - 函数长度适中

**6. 测试覆盖**（SHOULD）
- [ ] **单元测试**
  - 修改的核心逻辑有单元测试
  - 单元测试通过率100%
  - 边界条件有测试用例

- [ ] **集成测试**
  - 关键流程有集成测试
  - 多模块协作正常
  - 外部依赖mock测试

- [ ] **手动测试**
  - 正常流程测试通过
  - 异常流程测试通过
  - 用户场景测试通过

**7. 安全性**（SHOULD）
- [ ] **输入验证**
  - 所有用户输入都经过验证
  - 防止SQL注入
  - 防止XSS攻击

- [ ] **权限控制**
  - 访问权限正确
  - 操作权限正确
  - 数据权限正确

- [ ] **日志安全**
  - 不记录敏感信息
  - 日志脱敏处理
  - 日志访问控制

**8. 文档更新**（MAY）
- [ ] **代码注释**
  - 修改的关键逻辑有注释
  - 复杂算法有说明
  - 公共接口有文档

- [ ] **接口文档**
  - API文档更新
  - 参数说明准确
  - 返回值说明准确

- [ ] **用户文档**
  - 用户手册更新（如涉及功能变更）
  - 帮助文档更新
  - 更新日志记录

**详细验收标准**：见 [verification-checklist.md](../bug-fix-debugger/references/verification-checklist.md)

**智能体能力**：标准制定、验证清单生成

### 步骤15：验收检查

#### 15.1 逐项验收
按照验收标准逐项进行检查：

- **验收准备**：
  - 准备测试环境
  - 准备测试数据
  - 准备测试用例

- **执行验收**：
  - 按照验收标准逐项验证
  - 记录验收结果
  - 收集问题列表

- **问题处理**：
  - 评估问题严重程度
  - 决定是否需要修复
  - 修复后重新验收

#### 15.2 汇总修复的问题点清单
汇总本次修复的所有问题点：

**问题清单格式**：

| 问题编号 | 问题描述 | 严重程度 | 处理状态 | 验证结果 |
|---------|---------|---------|---------|---------|
| ISSUE001 | ... | 高/中/低 | 已修复/待处理 | 通过/不通过 |
| ISSUE002 | ... | 高/中/低 | 已修复/待处理 | 通过/不通过 |

#### 15.3 提出后续改进建议
基于本次修复经验，提出改进建议：

- **流程改进**：
  - 哪些流程可以优化
  - 哪些工具可以引入
  - 哪些规范需要完善

- **技术改进**：
  - 代码架构改进建议
  - 技术债务清理建议
  - 技术选型建议

- **团队改进**：
  - 技能培训需求
  - 知识分享建议
  - 团队协作优化

**智能体能力**：验收执行、总结复盘

## 输出物

完成本阶段后，应产出：
1. **执行记录**：任务执行情况、关键决策、异常处理
2. **验收报告**：验收标准逐项检查结果
3. **问题清单**：发现的所有问题及其处理状态
4. **验收结论**：是否通过验收，后续建议

## 注意事项

- **严格验收**：按照验收标准逐项检查，不放过任何问题
- **问题分类**：对发现的问题进行分类，区分是否需要立即修复
- **知识沉淀**：将本次修复的经验教训记录下来，形成知识库
- **持续改进**：基于验收结果，不断优化流程和方法

## 使用示例

### 示例1：简单BUG验收
**场景**：登录按钮无响应，修复后在拦截器中添加token刷新逻辑

**验收标准**：
- 功能验证：登录正常，token自动刷新
- 回归测试：不影响其他认证接口
- 性能评估：登录响应时间<2s

**验收结果**：
- 功能验证：✓ 通过
- 回归测试：✓ 通过
- 性能评估：✓ 通过

**验收结论**：✓ 通过

### 示例2：复杂问题验收
**场景**：系统响应慢，修复后优化了SQL查询并添加索引

**验收标准**：
- 功能验证：所有相关功能正常
- 回归测试：核心业务流程正常
- 性能评估：响应时间<300ms
- 数据完整性：数据准确无误

**验收结果**：
- 功能验证：✓ 通过
- 回归测试：✓ 通过
- 性能评估：✓ 通过（平均280ms）
- 数据完整性：✓ 通过
- 发现问题：写入性能下降15%

**问题处理**：
- 写入性能下降在可接受范围，记录为待优化项

**验收结论**：✓ 有条件通过（需后续优化写入性能）
