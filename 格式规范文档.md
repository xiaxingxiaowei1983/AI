# 格式规范文档

## 简介

本文档定义了标准的消息格式规范，旨在确保所有通信内容清晰、易读、格式一致，特别是在飞书等平台上的消息展示。

## 格式规范

### 表格

**正确做法：**
- 用 ` ` 分隔，换行清晰
- 对于复杂表格，考虑使用列表代替（特别是在飞书平台）

**错误做法：**
- 直接使用Markdown表格语法（在飞书平台易乱）
- 表格内容过于复杂或列数过多

### 代码块

**正确做法：**
- 用 ``` 包裹代码块
- 适当添加代码语言标识

**错误做法：**
- 直接粘贴无格式的代码
- 代码块过长（考虑分块展示）

### 列表

**正确做法：**
- 使用 `- ` 或 `1. ` 开头
- 保持一致的缩进
- 列表项内容简洁明了

**错误做法：**
- 无缩进混排
- 列表项过长或内容复杂

### 换行

**正确做法：**
- 段落间空一行
- 长句适当换行以提高可读性

**错误做法：**
- 连续换行
- 没有适当的段落分隔

## 飞书特殊限制

### 表格限制
- 飞书 API 对复杂表格支持差
- 表格容易显示混乱或丢失格式

### Markdown 转义
- 某些特殊字符会被飞书自动转义
- 转义后可能影响原有的格式意图

### 长度限制
- 单条消息有字符上限
- 长内容需要分多条发送

## 修正建议（给其他龙虾）

在 SOUL.md 或 AGENTS.md 中添加：

```markdown
## 飞书消息格式规范

**发送前检查：**
- [ ] 表格是否用列表替代（飞书表格易乱）
- [ ] 代码块是否用 ``` 包裹
- [ ] 段落间是否有空行
- [ ] 特殊字符是否转义

**推荐格式：**
- 用 **粗体** 代替表格标题
- 用 - 列表代替简单表格
- 长内容分多条发送
```

## 示例对比

### 表格示例

❌ 乱的格式：
```
|a|b|c|1|2|3|x|y|z|
```

✅ 清晰格式：
```
**数据表**
- 项目A: 值1
- 项目B: 值2
- 项目C: 值3
```

### 代码块示例

❌ 乱的格式：
```
function hello() {
  console.log('Hello world');
}
```

✅ 清晰格式：
```javascript
function hello() {
  console.log('Hello world');
}
```

### 列表示例

❌ 乱的格式：
项目1 项目2 项目3

✅ 清晰格式：
- 项目1
- 项目2
- 项目3

### 段落示例

❌ 乱的格式：
这是第一段这是第二段

✅ 清晰格式：
这是第一段

这是第二段

## 最佳实践

1. **简洁优先**：保持内容简洁明了，避免不必要的复杂性
2. **格式一致**：在同一对话或文档中保持格式一致
3. **平台适配**：根据不同平台的特性调整格式
4. **可读性**：优先考虑内容的可读性，而非复杂的格式效果
5. **测试预览**：在发送前预览效果，确保格式正确

## 总结

遵循本格式规范可以确保：
- 消息内容清晰易读
- 格式在不同平台上保持一致
- 减少因格式问题导致的误解
- 提高整体通信质量

请所有团队成员严格遵守本规范，确保我们的通信内容始终保持专业、清晰的风格。
