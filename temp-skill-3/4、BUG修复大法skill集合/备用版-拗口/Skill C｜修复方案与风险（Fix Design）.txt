
          
          
# Skill C｜修复方案与风险（Fix Design）

**一句话定义**  
在动手改之前，把“怎么修”压缩成 **A/B/兜底三案**，并补齐：改动边界｜风险｜验证｜回滚｜上线策略，避免“修了又坏、改了才发现验收口径不一致”。

## 适用场景（触发条件）
- Skill B 已有“当前最可能根因（含证据/置信度）”，准备进入修复
- 涉及上线、跨端兼容、改协议/数据结构/权限/配置/第三方依赖
- 修复一旦出错会扩大影响面，需要先把回滚与兜底想清楚

## 核心原则（强制）
- **先定验收口径（DoD）再定方案**
- 至少给 **2 个方案**（最快验证 vs 长期稳健），必要时给 **兜底方案**
- 每个方案必须配套：**验证路径｜回滚方式｜风险点｜影响面**
- 优先选择：**最小改动、可灰度、可回滚、可观测**
- 允许临时降级，但必须写清楚：**非最终修复 + 移除条件**

---

## 你要先提供的输入（短即可）
**最低必需**
1) 根因结论（来自 Skill B）：一句话 + 关键证据  
2) 验收标准（来自 Skill A）：通过/失败怎么判定

**可选但很重要**
3) 改动限制：哪些模块不能动？能不能加开关？能不能灰度？  
4) 风险偏好：更稳妥还是更快见效？  
5) 上线约束：是否必须今天止血？是否允许停机/重启？

## 高风险自动标记（命中任一条就必须有 Plan B/兜底）

## 输出要求（节省 token 的硬规矩）
