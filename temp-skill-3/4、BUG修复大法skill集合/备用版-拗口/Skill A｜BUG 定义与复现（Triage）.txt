

# Skill A｜BUG 定义与复现（Triage）
一句话定义
 把“别人说有问题/我觉得不对劲”变成一份能交给任何人开工的材料： 可复现、可验收、边界清晰、证据齐全、已止损（如需要） 。

## 适用场景（触发条件）
- 刚收到反馈，描述模糊：不知道是不是 BUG、在哪里、怎么复现
- 复现不稳定：时好时坏、换账号/换环境就不一样
- 影响面不明：不知道是单用户、部分用户、全量
- 线上告警/事故：需要先止损并快速对齐“问题口径”
- 第三方/权限/环境变量疑似相关：根因可能不在代码
## 核心原则（强制）
- 先把 “通过/失败标准” 写清楚，再谈修复
- 任何结论都要有 证据 （截图/日志/错误码/请求ID/时间点）
- 复现步骤必须 最小化 ：能用 3 步复现就不要写 10 步
- 如果影响线上体验，优先给 止损方案 （临时降级/开关/回滚），并标注“非最终修复”
## 你要先提供的输入（两档）
最低必需（没有这俩就无法开始）

1. 现象一句话：现在发生了什么？
2. 期望一句话：应该是什么？
建议补齐（越全越快定位） 3) 影响范围：谁受影响/频率/严重程度（P0/P1/P2）
 4) 环境信息：端（H5/小程序/APP/后台）+ 版本号 + 网络 + 账号类型
 5) 最近变更：最近是否发版/改配置/换依赖/改权限/换第三方
 6) 你手里已有证据：截图/录屏/日志/错误码/请求ID/时间点

# 输出模板（我将严格按此结构输出）
## 1. 问题定义（Problem Statement）
- 现象（用户可见）：
- 期望行为：
- 严重程度：P0 / P1 / P2
- 影响范围：哪些用户/哪些功能/频率（每次/偶发/仅首次）
## 2. 发生边界（Where / When）
- 会发生：列 3 条最关键条件（端/账号/入口/特定操作）
- 不发生：列 3 条对照条件（用于缩小范围）
- 时间窗口：首次出现时间/最近一次出现时间
## 3. 最小复现路径（Minimal Repro）
- 复现前置条件：账号/权限/开关/配置/数据准备
- 复现步骤（尽量 3–6 步，每步一句话）：

  2. …
  3. …
- 复现结果（你看到的失败信号）：
- 复现稳定性：必现 / 高概率 / 偶发（约 X%）

- 通过标准（客观可判断，1–3 条）：
- 失败标准（客观可判断，1–3 条）：

至少包含 4 类（能给多少给多少）：

## 输出要求（防 token、防空转的小规矩）
- 任何段落不确定就写“未知”，不要硬编
- 复现步骤超过 8 步，必须再压缩成最小路径
- 验收标准不得出现“应该/大概/感觉”，必须可判定
- 没有证据包也能输出，但必须明确“证据缺口”是什么
