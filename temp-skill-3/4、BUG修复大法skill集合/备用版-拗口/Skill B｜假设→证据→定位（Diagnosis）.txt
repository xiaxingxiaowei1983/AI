# Skill B｜假设→证据→定位（Diagnosis）

**一句话定义**  
用“假设驱动 + 证据验证”的方式，在最少 token 下把问题定位到：**是哪一类原因、在哪一段链路、下一刀该改什么**（而不是大段分析或把所有可能性都写完）。

## 适用场景（触发条件）
- 你已经有了 Skill A 的“问题定义 + 最小复现 + 验收标准”，但还不知道根因
- 修复反复回退：需要用证据把结论钉死
- 现象复杂：可能是代码/配置/权限/网络/第三方/缓存/数据导致
- 性能回退、数据异常、线上告警：同样适用（它本质是“问题定位法”，不是只修 BUG）

## 核心原则（强制）
- **一次只验证 2–5 个假设**：不搞大而全
- 假设必须写成“可验证句子”：能通过一次检查/脚本/日志对照来判定真伪
- 每个假设必须配套：**验证动作｜预期证据｜判定标准｜下一步分叉**
- 不许“凭感觉结论”：没有证据就保持未知，继续验证
- 优先排除“非代码因素”：配置/权限/环境变量/网络/第三方配额/回调

---

## 你要先提供的输入（越短越好）
**最低必需**
1) Skill A 的输出（至少包含：最小复现 + 验收标准 + 证据包里现有信息）

**可选但很有帮助**
2) 你最怀疑的 1–3 个方向（比如：权限/配置/第三方/版本/缓存/并发）  
3) 可用的观测手段：能不能看后端日志？能不能抓包？能不能跑最小脚本？  
4) 最近变更（有/无，是什么）

## 2) 假设池（只保留 Top 5）

- 配置 / 环境变量 / 开关

写法必须是“可验证句子”，例如：

## 验证动作清单（可直接复用的“快检菜单”）
你可以从这里挑动作塞进上面的“验证计划”，避免临场想不出来：

- **对照法**：同账号不同环境 / 同环境不同账号 / 同请求不同端点
- **最小脚本**：绕开前端，用脚本直打接口或直连第三方
- **请求构造核对**：host/path/query/header/body 是否符合预期
- **日志定位**：按时间点 + 请求ID/链路ID 找到服务端入口与错误分支
- **配置溯源**：实际运行时读到的配置值是什么（不是你以为的是什么）
- **缓存排除**：禁用缓存/换 key/强制刷新，看现象是否变化
- **灰度/版本排除**：不同版本/不同灰度组是否一致
- **网络排除**：换网络/关代理/换 DNS/检查证书与跨域报错
- **权限排除**：换管理员账号/检查控制台勾选项/检查 token 作用域

---

## 输出要求（节省 token 的硬规矩）
- 假设不超过 5 条；验证动作不超过 2 轮
- 不写长篇“可能原因大全”，只写“可验证 top 假设”
- 每条假设都必须能在 10 分钟内做一次判定；做不到就先拆成更小假设
- 结论必须带置信度；低置信度就明确“还缺哪条证据”
