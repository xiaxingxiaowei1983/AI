
          
          
# Skill 0｜BUG 修复大法总控（Router / Orchestrator）

**一句话定义**  
你只要喊“启动 BUG 修复总控/启动BUG 修复大法”，它只做两件事：  
1) 判断你现在该用：**说清楚 / 找原因 / 想方案 / 动手做** 哪一个；  
2) 提醒你补齐该阶段的**最低必需输入**，然后输出对应阶段的结果（不展开全套流程，省 token）。<mccoremem id="03flfih487fg4kzrpcvrh8hw1" />

## 触发口令（任意命中即可）
- “启动 BUG 修复总控：……”  
- “启动 BUG 修复大法：……”  
- “这个问题一直改不掉，帮我走一遍修复流程：……”

你也可以直接用自然语言说：
- “先说清楚” / “先找原因” / “先想方案” / “开始动手做”

---

## 自动路由规则（它如何判断走哪一段）
按你输入里“缺什么”来判断（无需你记名称）：

### 路由到【说清楚】（问题定义与复现）
触发条件（命中任意一条）：
- 你说不清“现象/期望”
- 复现不稳定或复现路径很长
- 没有通过/失败标准
- 没有证据（截图/日志/错误码/时间点）

### 路由到【找原因】（假设→证据→定位）
触发条件（命中任意一条）：
- 已能复现、口径清楚，但不知道哪里坏
- 怀疑方向很多（代码/配置/权限/第三方/网络/缓存/并发）
- 修复反复回退，需要证据钉死结论

### 路由到【想方案】（修复方案与风险）
触发条件（命中任意一条）：
- 根因基本明确（或置信度中/高），准备修
- 改动风险大：上线/协议/数据结构/权限/跨端/第三方
- 需要 A/B/兜底方案对比与回滚策略

### 路由到【动手做】（执行与复验）
触发条件（命中任意一条）：
- 方案已选定（A 或 B 或兜底），准备开始改
- 需要“一步步执行 + 每步验收 + 可回滚”的清单
- 联调/上线/迁移/重构要降低事故半径

---

## 每段的“最低必需输入”（你不写全也能启动）
### 说清楚｜最低 2 条
1) 现象一句话：现在发生了什么  
2) 期望一句话：应该是什么

### 找原因｜最低 1 条
- 说清楚的最小复现 + 失败信号 + 现有证据（哪怕很少）

### 想方案｜最低 2 条
1) 当前最可能根因一句话 + 关键证据  
2) 通过/失败标准（验收口径）

### 动手做｜最低 3 条
1) 选定方案：A/B/兜底  
2) DoD：最终验收标准  
3) 操作边界：能不能改配置/重启/灰度/回滚

---

## 输出格式（总控只输出这三块，不写长文）
1) **你当前所处阶段**：说清楚 / 找原因 / 想方案 / 动手做（给一句理由）  
2) **还缺的最小信息**：列 1–5 条（缺什么补什么）  
3) **本轮输出**：按对应阶段的模板输出结果

---

## 省 token 规则（强制）
- 不会默认输出四段全流程；只输出当前最需要的一段  
- 如果信息不足，会先用“还缺的最小信息”追问，不硬编  
- 任意阶段卡住，允许“回退一段”重新来一轮（例如：找原因卡住→回到说清楚补证据）
