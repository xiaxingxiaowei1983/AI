# Skill 1：一句话定位 & 边界（Idea Freeze Spec）

## 目标
在你写 PRD、做技术方案、开始开发之前，先把产品的“方向盘”定死：
- 这产品给谁用？
- 解决什么问题？
- 用什么方式解决？
- 哪些不做（边界）？
- 什么算“上线成功”（可验收）？

> 这一步做不好，后面会出现：需求膨胀、反复改方向、做完发现没人用。

## 何时调用（触发条件）
- 你有一个新点子（例如“盲眼小郎中”）
- 你准备开始写 PRD/招人开发/套壳
- 你发现团队对“到底做什么”理解不一致时

## 输入（固定表单，一次填完）
> 不确定也要写“暂定”，后续只能升版本修改。

### A. 一句话定位（必填）
按这个句式填空：
- **给谁（目标用户）**：____
- **在什么场景（触发时刻）**：____
- **解决什么痛点（具体问题）**：____
- **我们怎么解决（核心方式）**：____
- **带来什么结果（可感知收益）**：____

合成一句话（模板）：
> “面向【目标用户】，在【场景】下，解决【痛点】，通过【方式】，让用户获得【收益】。”

### B. 价值主张（必填）
- 用户为什么要用你而不是不用（1 句）：____
- 用户为什么要用你而不是用竞品（1 句）：____

### C. 最小闭环（MVP 闭环）（必填）
用 3 步描述“用户第一次使用就能完成的闭环”：
- Step1（用户做什么）：____
- Step2（系统做什么）：____
- Step3（用户得到什么结果）：____

### D. 不做清单（边界）（必填，至少 5 条）
这些是“明确不做”的，防止范围膨胀：
- 不做 1：____
- 不做 2：____
- 不做 3：____
- 不做 4：____
- 不做 5：____

### E. 风险假设（必填，选 3 个）
- 假设 1（如果不成立会失败）：____
- 假设 2：____
- 假设 3：____

### F. 上线成功标准（必填，可量化）
至少写 5 条，用“能否判定”的句子：
- S1：____（例：用户能完成一次完整闭环）
- S2：____（例：录音授权成功率≥__% 或 提供无录音降级）
- S3：____（例：关键页面崩溃率<__%）
- S4：____（例：核心链路 5 条冒烟全绿）
- S5：____（例：有至少 __ 个真实用户连续使用 __ 天）

### G. 版本号（必填）
- Idea Freeze Spec 版本：v1
- 变更规则：只有 v1→v2 这样升版本，禁止口头改方向

## 冻结规则（不可变规则）
- R1：一句话定位与最小闭环没定死前，不进入大量开发。
- R2：不做清单是强约束，新增需求默认先进入“不做”，除非升级版本。
- R3：所有讨论必须回到“上线成功标准”，不能靠感觉争论。

## 验收点（可量化）
- A1：一句话定位填满，且不含“提升/优化/更好”这种模糊词
- A2：最小闭环 3 步写清楚（用户动作可执行）
- A3：不做清单 ≥ 5 条
- A4：风险假设 ≥ 3 条且可验证
- A5：上线成功标准 ≥ 5 条且可判定（能打勾/打叉）

## 输出格式（固定：Idea Freeze Spec 结果单）
- Idea Freeze Spec：PASS / FAIL
- 版本：v1
- 一句话定位：____
- 最小闭环（3步）：1)___ 2)___ 3)___
- 不做清单（≥5）：___
- 风险假设（≥3）：___
- 上线成功标准（≥5）：___
- 若 FAIL：前三定位路径：
  - P1：一句话定位是否仍模糊（谁/场景/痛点没写清）
  - P2：不做清单是否不足（范围在膨胀）
  - P3：成功标准是否不可验收（都是“感觉更好”）

## 完成标准（本 skill 完成的判定）
- 你产出一份 v1 的结果单
- 团队成员看完能一致回答：我们做什么、先做什么、不做什么、何时算成功
