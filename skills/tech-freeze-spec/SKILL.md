# Skill 4：技术方案冻结（Tech Freeze Spec）

## 目标
把技术方案写成“唯一标准答案”，并把最容易返工的口径一次性定死：
- 形态：小程序 / H5 / 两者
- 入口：域名、API、WS/Socket 路径、协议（https/wss）
- 数据格式：尤其音频（H5 ArrayBuffer vs 小程序 base64+format）<mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" />
- 平台约束：小程序白名单、权限、隐私
- 后端边界：模块、依赖、数据存储
- 验收与回滚：如何证明上线成功、失败怎么止血

> 你们之前最痛苦的返工，很多来自“入口口径没冻结、环境漂移、通道混用”。这一步做扎实，后面省很多 token 和崩溃。

## 何时调用（触发条件）
- Skill 3（MVP 切割）完成后，准备开始开发/套壳之前（必跑）
- 你要确定发布形态和上线方式时
- 出现这些现象时（说明口径没冻结）：
  - “本地通线上不通”
  - “接口通但实时/音频不通”
  - “改来改去都差一点”

## 输入（固定表单）
- 产品名：
- MVP 版本：v0.1
- 目标形态：小程序 / H5 / 两者
- 核心链路（写 3 条）：____
- 是否启用实时：是/否
- 是否涉及录音/音频：是/否
- 你们现有壳技术栈（可选）：uni-app / NestJS / 其他

## 前置约束（硬规则）
- 所有“入口口径”（域名/API/WS/路径/协议）必须写死，不允许口头改
- 所有“数据格式”（音频字段/编码/format）必须写死，不允许双方各猜各的
- 变更必须升版本（v1→v2），否则必返工

## 冻结清单（按章节填写）
### 1）形态与端侧能力（必填）
- 发布形态：小程序/H5/两者
- 端侧关键能力：
  - 登录：____（例如手机号/微信登录/匿名）
  - 实时：____（Socket.IO / 原生 WS）
  - 录音：____（需要/不需要）
- 端侧不可控约束：
  - 小程序：白名单/审核/权限隐私
  - H5：HTTPS/Mixed Content/反代/缓存

### 2）入口冻结（最关键，必填）
> 写出“用户/客户端实际访问的完整入口”。

- H5（如有）：
  - 站点 URL：https://____
  - API 策略：`/api` 同域反代 / 独立域名
  - 实时：`/socket.io/`（若 H5 用 Socket.IO）<mccoremem id="03fl75psdph7oiq63zur4f5vg" />
- 小程序（如有）：
  - API Base（必须 https）：https://____
  - WSS（必须 wss）：wss://____ + 路径 `/ws`（你们常用）<mccoremem id="03fl75psdph7oiq63zur4f5vg" />
  - 微信白名单口径：request/socket/upload/download/业务域名（如 web-view）

- 后端真实运行：
  - 内部端口：____（例 3006）<mccoremem id="03fl75psdph7oiq63zur4f5vg" />
  - 对外方式：Nginx/网关反代到 443（推荐）
  - 健康检查：/health（或无）

### 3）实时与消息协议冻结（如启用实时）
- 实时类型：Socket.IO / 原生 WS
- 路径：/socket.io/ 或 /ws（写死）
- 消息协议（写死 3 件事）：
  - message 基本结构：例如 `{type, payload, ts}`（按你们实际写）
  - 心跳/重连策略：例如 10s 心跳、断线重连
  - 错误码/错误消息：例如 `type:'error', code, msg`

### 4）音频数据格式冻结（如涉及音频）
> 这块是你们最容易踩坑的地方，必须写得非常具体。<mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" />

- H5 上行格式：ArrayBuffer/TypedArray（写死字段名）
- 小程序上行格式：base64 + format（写死字段名，例如 `{type:'audioData', audio, format}`）
- format 允许值：pcm/wav/mp3（按你们真实支持写）
- 最大包大小/分片策略：是否分片、每片大小
- 后端兼容要求：必须同时兼容 H5 与小程序两套格式

### 5）权限与隐私冻结（小程序尤其必填）
- 必需权限：录音/相机/相册/位置（勾选）
- 隐私政策链接：https://____
- 数据收集说明（三句话）：
  - 收集什么：
  - 为什么收集：
  - 怎么处理：
- 拒绝授权兜底：用户拒绝录音时产品如何降级（必须写）

### 6）后端模块边界与依赖（必填）
- 模块边界（写清楚 3–6 个模块）：
  - 认证/用户
  - 会话/对话
  - 实时/WS
  - 音频处理
  - 存储/上传
- 依赖清单（只写名称，不写密钥）：
  - DB：MySQL/Redis/…
  - 对象存储：COS/OSS/…
  - 第三方语音：ASR/TTS/…
- 环境变量权威来源：.env / 进程注入 / Docker env（只能选一个）

### 7）上线验收与回滚策略（必填）
- 上线前置校验门：用你们已生成的 H5/小程序 Preflight Gate
- 冒烟验收：用你们已生成的 Smoke Test（H5/小程序对应）
- 回滚策略：用你们已生成的 Rollback Playbook（H5/小程序对应）
- 上线成功标准（至少 5 条）：____

## 冻结规则（不可变规则）
至少 10 条，给你模板：
- R1：入口（域名/API/WS/路径/协议）冻结，不口头改
- R2：音频字段/format 冻结，不口头改
- R3：端侧通道不混用（H5 Socket.IO vs 小程序 /ws）<mccoremem id="03fl75psdph7oiq63zur4f5vg" />
- R4：env 权威来源唯一
- R5：所有密钥不进前端不进仓库
- R6：发布前必须跑前置校验门与冒烟
- R7：失败即止损（不长时间试错），按定位路径或回滚
- R8：变更必须升版本并同步文档
- R9：数据/存储必须隔离（套壳项目）
- R10：真机/线上验证优先于模拟器判断（小程序）

## 验收点（可量化）
- A1：入口冻结完整（能写出完整 URL/路径）
- A2：实时协议与音频格式写到“字段级”且可实现
- A3：权限/隐私材料齐全（小程序可提审）
- A4：后端模块与依赖清晰，env 清单明确
- A5：验收与回滚策略明确（用已生成 skill）

## 输出格式（固定：Tech Freeze Spec 结果单）
- Tech Freeze Spec：PASS / FAIL
- 版本：v1
- 形态：小程序/H5/两者
- 入口冻结：____
- 实时协议：____
- 音频格式：____
- 权限隐私：____
- 后端边界与依赖：____
- 上线验收与回滚：____
- 若 FAIL：前三定位路径：
  - P1：入口是否未写到完整 URL/路径
  - P2：音频/实时是否未写到字段级
  - P3：权限隐私/白名单是否缺失（小程序）

## 完成标准（本 skill 完成的判定）
- 你产出 v1 的结果单
- 团队能按这份文档直接进入“套壳与开发”，且不需要反复问“到底连哪里/格式是什么”
