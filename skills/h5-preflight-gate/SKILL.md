# Skill 2：发布前置校验门（Preflight Gate）

## 目标
在你进入“正式上线/联调”之前，用最短路径做一次**一票否决**的预检：
- 通过：进入下一步（反代/部署/验收）
- 不通过：立刻停止，按固定定位路径处理，不进入长对话排查

> 这个 skill 的价值：减少返工、减少“试试改这个/再改那个”、减少 token 消耗。

## 何时调用（触发条件）
- 每次 H5 准备发到生产（prod）之前
- 每次更换域名、证书、反代路径、后端端口之后
- 出现这些现象时：
  - “本地通线上不通”
  - “接口通了但 Socket/实时不通”
  - “我改了但没生效”
  - “线上拿到旧包/旧配置”

## 输入（固定表单）
- 项目路径：
- 目标环境：`dev` / `staging` / `prod`
- H5 站点 URL（含协议）：`https://____`
- API 策略：`/api` 同域反代 / `独立API域名`
- 后端真实端口：`____`（例：3006）
- 是否启用实时：`是/否`
- 实时类型：`Socket.IO` / `原生WS`
- 实时路径：`/socket.io/` 或 `/ws`
- 是否启用 HTTPS：`是/否`（推荐：是）

## 前置约束（硬规则）
- **只要任意一条 Fail，本次发布流程停止**，先修复再重新跑本 skill
- 本 skill 只输出：Pass/Fail + 证据 + “前三定位路径”
- 不做“猜测式长排查”，不展开到下一个 skill

## 预检清单（5 条硬检查）
每条都必须输出：`结果(✅/❌) + 证据 + 下一步`

### Gate 1：残留扫描（入口一致性）
**检查目标**：杜绝 `localhost/127.0.0.1/旧域名/旧端口` 残留导致线上请求跑飞  
- 证据要求（任选其一即可）：
  - 前端构建产物或源码扫描结果：0 命中（关键词：`localhost`、`127.0.0.1`、`http://`、旧域名、`3005/3006/300x`）
  - 线上浏览器 Network：所有请求都命中预期域名/路径（与 Freeze Spec 一致）
- Fail 的前三定位路径：
  1) 前端配置入口是否唯一（是否散落多个 baseURL）
  2) 是否存在旧包缓存（CDN/浏览器/服务器静态目录）
  3) 构建时注入 vs 运行时注入是否搞反

### Gate 2：环境变量齐全（Env Contract）
**检查目标**：避免“我改了 env 但服务没读到 / 缺变量导致隐性异常”  
- 证据要求：
  - 列出“必须变量清单”（最少 8 条）：端口、数据库、鉴权密钥、第三方 key、回调域名、WS 配置等
  - 实际生产环境逐条对照：无缺失、无空值、无明显错值（例如把 dev 域名填进 prod）
- Fail 的前三定位路径：
  1) 服务实际读取的 env 来源（文件/pm2/systemd）
  2) 是否存在多份 `.env` 或覆盖优先级问题
  3) 重启是否真正生效（看启动日志关键字）

### Gate 3：Nginx 配置可用（Syntax + Path）
**检查目标**：避免“配置写了但没加载/路径不匹配/WS Upgrade 丢失”  
- 证据要求（至少满足两条）：
  - Nginx 配置语法检查通过（输出 ok）
  - `https://<H5域名>/` 访问返回 200（或 304）
  - `/api/...` 命中反代（访问日志能看到）
  - 若启用实时：`/socket.io/` 或 `/ws` 路径能命中并完成握手/升级
- Fail 的前三定位路径：
  1) 站点 server_name/证书是否对应域名
  2) `/api` 前缀是否与前端一致（多/少一个斜杠都可能错）
  3) 实时通道是否配置了 Upgrade（尤其是 Socket.IO）

### Gate 4：后端端口监听（Server Listening）
**检查目标**：避免“反代都对，但后端没起来/端口不对/只监听内网导致不可达”  
- 证据要求：
  - 后端进程存在（pm2/systemd 状态为 running）
  - 端口监听存在（端口=输入表单的端口）
  - 轻量接口能直连（从服务器本机访问 127.0.0.1:端口 返回 200）
- Fail 的前三定位路径：
  1) 启动命令是否用的 prod 配置
  2) 端口是否被占用或被改写
  3) 日志里是否有启动失败/连接数据库失败等关键错误

### Gate 5：实时路径可达（WS/Socket Readiness）
**检查目标**：避免“接口通了但实时不通”反复联调  
- 证据要求（按类型二选一）：
  - Socket.IO：前端能建立连接，并能完成一次收发（或服务端看到 connect 日志）
  - 原生 WS：wss 握手成功，并能收到一条服务器响应/心跳
- Fail 的前三定位路径：
  1) 路径是否正确（`/socket.io/` vs `/socket.io`，`/ws` 是否一致）
  2) 协议是否一致（https 页面必须 wss）
  3) Nginx 是否放行升级与必要 header

## 输出格式（固定，直接复制到发布记录）
- Preflight 结论：PASS / FAIL
- 环境：dev/staging/prod
- Freeze Spec 版本号：v__
- Gate1 残留扫描：✅/❌（证据：___）
- Gate2 Env 齐全：✅/❌（证据：___）
- Gate3 Nginx 可用：✅/❌（证据：___）
- Gate4 端口监听：✅/❌（证据：___）
- Gate5 实时可达：✅/❌（证据：___）
- 如果 FAIL：本次停止；优先定位路径：P1___ / P2___ / P3___

## 完成标准（本 skill 完成的判定）
- 你拿到了一个明确结论 PASS/FAIL
- 每个 Gate 都有“证据”（不是主观判断）
- FAIL 时有“前三定位路径”，且流程已停止（不进入下一 skill）
