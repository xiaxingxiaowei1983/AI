# Skill 7：回滚 Playbook（Rollback Playbook）

## 目标
当上线后出现故障时，用最短路径把系统恢复到“上一个可用版本”，并用固定验收点证明已恢复。
- 不追根因、不边修边试（那是复盘阶段做的事）
- 先恢复可用性，再做定位与修复

## 何时调用（触发条件）
满足任意一条就触发回滚：
- Smoke Test 判定 FAIL（尤其是 Case4/Case5/Case6/Case7 失败）
- 线上核心链路不可用（登录/核心接口/关键页面不可用）
- 线上出现大面积 5xx 或实时通道不可用（Socket.IO/WS 连不上）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 发布后出现严重兼容问题（白屏、资源大面积 404、Mixed Content 爆炸）
- 你无法在 10–15 分钟内定位并修复（此时继续试错只会扩大损失）

## 输入（固定表单）
- 项目名：
- 环境：`staging` / `prod`
- 本次发布版本标识：`new = ____`
- 上一个稳定版本标识：`stable = ____`
- 发布类型：`前端` / `后端` / `全量`
- 是否启用 CDN：是/否
- 是否启用实时：是/否（Socket.IO/WS）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 负责人：
- 通知渠道（群/电话/工单）：`____`

## 回滚前硬规则（防止“越回越乱”）
- RR1：先“冻结发布”——停止继续上传新包/继续改配置
- RR2：只允许回滚到“明确验收过的稳定版本 stable”，不允许凭感觉选版本
- RR3：回滚顺序固定：**先前端入口（影响最大）→ 再后端（若必要）→ 最后配置/反代（谨慎）**
- RR4：回滚动作必须可记录：每一步都写清“做了什么、证据是什么”

## 回滚策略选择（先判断走哪条）
### Strategy A：只回滚前端（H5 入口问题最常见）
适用：
- 白屏/静态资源 404
- 入口页缓存导致新旧混乱
- 前端请求打错域名/路径
- Mixed Content

### Strategy B：只回滚后端（接口/实时/依赖问题）
适用：
- 5xx/接口不可用
- 实时连接失败（且确认反代没变）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- env 变更导致运行异常
- 数据库迁移/依赖变更引发故障

### Strategy C：全量回滚（前后端一起回）
适用：
- 版本强绑定（前端新接口/后端新协议）
- 多处同时改动，无法快速判定故障面

## 回滚步骤（按固定顺序执行）
> 下面每一步都要求：✅/❌ + 证据

### Step 0：发布阻断与公告
- 操作：
  - 在发布群发“开始回滚”的公告（包含 stable 版本）
  - 记录开始时间
- 通过标准：所有相关人知道“当前正在回滚，暂停发布”

### Step 1：前端回滚（H5 静态资源）
- 操作（选择你们真实方式其一）：
  - 方式1：把站点指向上一个 dist 目录（stable）
  - 方式2：替换静态目录为 stable 产物
  - 方式3：切换 Nginx 指向 stable 目录（谨慎，避免改错站点）
- CDN（若启用）：
  - 至少刷新 `index.html`（入口页）
- 通过标准（必须全满足）：
  - 首页能打开
  - 关键静态资源 404=0
  - 无痕模式能拿到 stable 版本（防缓存）
- 证据：无痕访问截图 + Network 关键资源状态

失败时前三定位路径：
1) CDN 缓存未刷新（入口页最关键）
2) 服务器静态目录未真正替换
3) Nginx 站点指向错目录/错 server_name

### Step 2：后端回滚（可选）
- 操作：
  - 方式1：切回 stable 版本构建物/镜像
  - 方式2：切回 stable 代码并重启服务（必须确认 env 不漂移）
  - 方式3：切回 stable 的运行配置（pm2/systemd/Docker）
- 通过标准（至少三条）：
  - 服务进程 running
  - 本机直连轻量接口 200（127.0.0.1:端口）
  - 通过 Nginx `/api` 访问轻量接口 200
- 证据：状态输出 + 轻量接口返回

失败时前三定位路径：
1) 实际读取的 env 来源是否正确（是否还在读 new 的配置）
2) 端口监听是否正确
3) 依赖是否恢复（DB/Redis/第三方）

### Step 3：实时通道确认（若启用）
- 操作：
  - Socket.IO：确认 `/socket.io/` 路径仍可握手/连接 <mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
  - 原生 WS：确认 `/ws` wss 握手成功 <mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 通过标准：连接建立 + 完成一次收发
- 证据：前端日志/后端 connect 日志关键字

失败时前三定位路径：
1) Nginx Upgrade 配置是否被改动
2) https→wss 协议是否一致（Mixed Content）
3) 后端实时服务是否启动/端口是否正确

### Step 4：最小冒烟复验（必做）
- 操作：只跑 Smoke Test 的关键 4 条（节省时间）：
  - 首页可打开
  - 轻量 API 200
  - 关键业务链路 1 条通过
  - 实时可用（若启用）
- 通过标准：这 4 条全绿才算回滚完成
- 证据：Smoke Test 复验记录

### Step 5：回滚完成公告 + 保留证据
- 操作：
  - 在群里发“回滚完成”公告（stable 版本 + 当前状态）
  - 记录结束时间
  - 把本次 new 版本标识、故障现象、触发回滚原因写入发布记录单
- 通过标准：所有人统一口径：系统已回到 stable

## 输出（固定：回滚记录单）
- Rollback：PASS / FAIL
- 项目 / 环境：
- new 版本标识：
- stable 版本标识：
- 触发原因（选择）：白屏/5xx/实时不可用/Mixed Content/缓存/其他
- 执行人：
- 开始时间 / 结束时间：
- Step1 前端回滚：✅/❌（证据：___）
- Step2 后端回滚：✅/❌（证据：___）
- Step3 实时确认：✅/❌（证据：___）
- Step4 复验：✅/❌（证据：___）
- 结论：已恢复 / 未恢复（若未恢复，下一步：升级为紧急故障处理流程）

## 完成标准（本 skill 完成的判定）
- 你能证明系统已回到 stable（有证据，不靠口头）
- 关键 4 条复验全绿
- 发布被阻断、公告同步到位
