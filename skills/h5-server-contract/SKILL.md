# Skill 4：后端上线运行契约（Server Runtime Contract）

## 目标
把后端在生产环境的运行方式写成一份“契约”，让团队不再靠记忆上线：
- **服务怎么启动、怎么重启、怎么确认生效**
- **读哪份 env、监听哪个端口**
- **日志在哪里、出错先看哪里**
- **健康检查是什么、什么算上线成功**
- **回滚怎么做**

> 你们这次“改了但没生效/端口漂移/环境变量混乱/重启方式不一致”的痛点，本质是没有运行契约。

## 何时调用（触发条件）
- 第一次把后端部署到生产环境（prod）
- 更换服务器/更换部署方式（pm2 ↔ systemd ↔ Docker）
- 改了 `.env`、端口、域名、反代后，担心“没生效”
- 出现以下现象：
  - 502/504（上游不可达/超时）
  - 服务看似在跑，但接口不通
  - 重启后行为不一致（像还在读旧配置）

## 输入（固定表单）
- 项目名：
- 目标环境：`dev` / `staging` / `prod`
- 后端框架：`NestJS`（或其他）
- 部署方式：`pm2` / `systemd` / `Docker` / `其他`
- 后端端口：`____`（例如 3006）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 监听地址：`127.0.0.1` / `0.0.0.0`
- env 来源（必须明确一种为主）：
  - A：`server/.env` 文件
  - B：pm2/systemd 注入环境变量
  - C：Docker env / secret
- 健康检查路径：`/health`（若没有就填 `无`）
- 日志方式：`stdout/stderr` / `文件` / `云日志`
- 数据库/外部依赖（只列名称，不写密钥）：`MySQL/Redis/对象存储/第三方API/...`

## 契约规则（不可变规则）
- R1：生产环境只允许一个“权威 env 来源”（输入表单选的那种），禁止多份 env 互相覆盖导致漂移。
- R2：生产端口固定（例如 3006），端口变更必须“升版本 + 同步 Freeze Spec + 同步 Nginx”。<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- R3：服务必须可被守护（进程挂了能拉起），禁止手动 `node xxx.js` 裸跑。
- R4：每次发布必须可验证“已生效”（必须有证据：健康检查/启动日志关键字/版本号）。
- R5：日志必须可追溯（至少能查到：启动成功、请求进入、报错栈、WS/Socket 连接情况）。
- R6：对外入口只通过 Nginx/网关（如适用），后端真实端口不直接暴露公网（降低风险与故障面）。

## 过程（执行步骤）
### Step 1：固定“端口与监听”
- 写死：端口 = 输入表单的端口
- 写死：监听地址 = 输入表单的监听地址（建议生产反代场景用 127.0.0.1）

输出（写入契约结果单）：
- 端口：
- 监听地址：
- 对外访问方式：`Nginx 反代` / `直接暴露`（推荐反代）

### Step 2：固定“env 来源与必备变量清单”
- 明确：env 从哪里读（A/B/C 三选一）
- 输出“必备变量清单”（只写键名，不写值），最少包含：
  - 端口相关（PORT 等）
  - 数据库连接（HOST/USER/DB 等）
  - 鉴权密钥（JWT/SESSION 等）
  - 跨域/回调域名（若有）
  - 第三方服务配置（若有）
  - 实时通道相关开关（若有）
  - 文件/对象存储配置（若有）

验收点：
- 缺失任何必备变量都应当让服务启动失败或给出明确错误（不能“悄悄用默认值”）

### Step 3：固定“启动/重启/查看状态”的唯一方式
- 部署方式选择其一并写死：
  - pm2：启动命令、重启命令、查看状态命令
  - systemd：service 名称、start/restart/status/journal 命令
  - Docker：容器名、restart policy、查看日志命令

输出（写入契约结果单）：
- 启动方式：
- 重启方式：
- 查看状态方式：
- 查看日志方式：

### Step 4：定义“健康检查与上线成功标准”
- 若已有健康检查：写死路径（例如 `/health`）
- 若没有：先定义一个“最轻量可用接口”作为临时健康检查（例如返回版本/时间的接口）

上线成功（必须满足全部）：
- S1：服务进程状态为 running
- S2：本机直连 `127.0.0.1:端口` 的轻量接口返回 200
- S3：经 Nginx 反代后对外访问接口返回 200
- S4：关键业务接口至少 1 条通过
- S5：若启用实时：连接/握手成功（从后端日志能看到 connect/握手证据）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>

### Step 5：定义“发布后快速定位路径（前三优先）”
出现 502/504/接口不通/实时不通，优先按顺序定位：
- P1：后端进程是否在跑 + 是否监听正确端口（最常见）
- P2：后端日志：启动错误/依赖连接错误/鉴权错误
- P3：Nginx 日志：是否把请求转到了正确上游、是否超时、实时是否 Upgrade

- 必备变量键名清单（不含值）：
- 健康检查路径：

- 状态查看方式：
- 日志查看方式：
- 上线成功标准：S1~S5
- 失败定位路径：P1~P3

## 验收点（可量化）
- A1：重启后服务自动恢复（守护生效）
- A2：本机直连轻量接口 200
- A3：反代访问轻量接口 200
- A4：改 env 后重启，行为确实变化且有证据（日志/返回内容/版本号）
- A5：出现错误时能在 5 分钟内定位到“进程/反代/配置/依赖”四类之一（不靠猜）

## 完成标准（本 skill 完成的判定）
- 你能拿出一份“运行契约结果单 v1”
- 团队成员按结果单操作不会出现“各自一套启动方式/各自一套 env”
- 每次发布后都能用同一套证据证明：服务已生效、可用、可定位
