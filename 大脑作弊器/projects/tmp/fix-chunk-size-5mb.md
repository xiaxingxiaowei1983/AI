# v1.1 分片上传修复说明

## 问题描述
用户上传12MB PDF文件时，出现413错误（Payload Too Large），导致部分分片上传失败。

### 错误详情
```
文件大小：12,512,644 字节（约11.9MB）
分片策略：2个分片（每个约6MB）
错误：分片1上传成功，分片2返回413错误
```

### 日志分析
```
❌ 分片 1/2 上传失败: SyntaxError: Unexpected token 'R', "Request En"... is not valid JSON
POST /api/upload-chunk 413 (Payload Too Large)
```

## 根本原因

### 1. FormData额外开销
FormData除了文件数据本身外，还有额外的元数据：
- **Boundary分隔符**：每个字段前后都有分隔符
- **Content-Disposition headers**：每个字段的属性信息
- **文件名和MIME类型**：额外的文本数据

### 2. 实际请求体计算
```
分片大小：6MB
FormData开销：约1-2MB（boundary + headers + fileId等参数）
实际请求体：7-8MB

但Coze外网网关限制可能是10MB，当分片接近10MB时，FormData总大小可能超标
```

### 3. 风险场景
- 文件大小接近10MB时，单个分片可能接近8-9MB
- 加上FormData开销，实际请求体可能超过10MB限制
- 导致413错误

## 解决方案

### 修改1：减小分片大小（已修复）
```javascript
// 旧设置
const CHUNK_SIZE = 10 * 1024 * 1024; // 10MB

// 新设置
const CHUNK_SIZE = 5 * 1024 * 1024; // 5MB
```

### 修改2：优化错误处理（已修复）
```javascript
// 添加413错误特殊处理
if (response.status === 413) {
    throw new Error('分片太大，请稍后重试（服务器正在调整配置）');
}
```

## 性能影响分析

### 优点
✅ **更安全**：避免413错误
✅ **更稳定**：适应各种网络环境
✅ **更可靠**：单个分片失败影响更小

### 缺点
❌ **请求次数增加**：分片数量翻倍
❌ **并发控制重要**：需要合理控制并发数（当前设置为3）

### 优化建议
1. **智能分片**：根据文件大小动态调整分片大小
   - 小文件（<5MB）：不分割
   - 中文件（5-15MB）：5MB分片
   - 大文件（>15MB）：8MB分片（在安全范围内最大化效率）

2. **自适应并发**：根据网络速度调整并发数
   - 慢速网络：1-2个并发
   - 快速网络：3-5个并发
