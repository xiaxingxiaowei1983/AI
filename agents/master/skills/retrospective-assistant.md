# 复盘助手

## 适用场景（触发条件）
- 用户说"帮我复盘"、"分析一下这个问题"、"系统性总结一下"
- 项目结束后需要进行经验沉淀
- 问题反复出现需要找出根因
- 需要从事件中提炼可复用的经验

## 总体原则
- **逻辑闭环**：从问题定义到沉淀复用，形成完整闭环
- **可验证**：每个结论都需要证据支撑
- **可复用**：最终输出可迁移的经验、模型、模板
- **不空泛**：基于事实，不确定的标注"待验证"

## 输出格式（必须严格按10步输出）

### 第一步：问题定义
- 这次事件或问题的本质是什么？
- 核心矛盾在哪里？
- 成功状态是什么？

### 第二步：目标与假设
- 当初的目标是什么？
- 显性与隐性目标分别是什么？
- 成功的判断标准是什么？

### 第三步：现状与差距
- 当前真实情况如何？
- 与目标的关键差距在哪里？
- 哪些是最重要的差距？

### 第四步：原因分析
- 提出可能原因（MECE原则：相互独立、完全穷尽）
- 对原因进行优先级排序
- 选择最关键的进行验证

### 第五步：证据与验证
- 基于数据、事实、行为或日志进行验证
- 哪些假设成立？哪些不成立？
- 不确定的标注为"待验证"

### 第六步：策略与对策
- 可能的解决路径有哪些？
- 哪些最优？
- 优先级如何？

### 第七步：风险与不确定性
- 可能出现的风险是什么？
- 哪些情况可能导致失败？
- 应对策略是什么？

### 第八步：行动与实验
- 下一步具体行动是什么？
- 如何用最小成本验证？

### 第九步：结果复核
- 如何判断是否成功？
- 验收标准是什么？

### 第十步：沉淀与复用
- 这次可以沉淀哪些原则、模型、模板或SOP？
- 哪些经验可以迁移到其他场景？
- 下次遇到类似情况，先做哪3件事？

## 分析方法工具箱（按需选择）

### 5Why分析法
适合：单链路因果关系追踪
用法：连续追问5次"为什么"，直到找到根因

### MECE原则
适合：多分支可能性排查
用法：确保原因分类"相互独立、完全穷尽"

### 假设驱动法
适合：信息不完整时的推理
用法：提出假设 → 设计验证 → 确认/推翻

### 鱼骨图
适合：多因素复杂问题
用法：从人、机、料、法、环、测六个维度分析

### PDCA循环
适合：需要持续改进的场景
用法：Plan → Do → Check → Act

### DMAIC
适合：需要建立长期稳定方案
用法：Define → Measure → Analyze → Improve → Control

## 证据采集清单
- 数据指标（量化）
- 日志/记录（可追溯）
- 用户反馈（可引用）
- 关键事件时间点
- 对比参照（A/B对照）

## 与其他Skill的关系
- **BUG修复大法**：技术BUG修复专用，复盘助手是通用分析框架
- **复盘报告**：复盘助手侧重分析过程，复盘报告侧重输出正式文档
- 可串联使用：先用复盘助手分析，再用复盘报告生成正式文档

## 示例输出

### 问题定义
- 本质：项目延期交付，核心矛盾是需求变更频繁与固定排期的冲突
- 成功状态：按时交付核心功能，非核心功能可迭代

### 原因分析（5Why）
1. 为什么延期？→ 需求变更频繁
2. 为什么变更频繁？→ 前期需求调研不充分
3. 为什么调研不充分？→ 时间紧迫，跳过了用户访谈
4. 为什么跳过用户访谈？→ 项目启动时未设立需求冻结机制
5. 根因：缺乏需求管理流程和冻结机制

### 沉淀与复用
- 模型：需求变更影响评估模型
- SOP：项目启动需求冻结流程
- 下次先做：1) 用户访谈 2) 需求冻结签字 3) 变更影响评估
