# 复盘报告

## 适用场景（触发条件）
- 用户说"写一份复盘报告"、"输出复盘文档"、"生成复盘报告"
- 需要向上级/客户/第三方提交正式复盘文档
- 项目结束后需要归档复盘材料
- 需要将零散素材整理成结构化报告

## 硬性要求
1. **只基于素材**：不允许编造；不确定就标注【待确认】并给出需要补充的信息清单
2. **写的是"复盘报告"**：不是建议清单，必须有背景、目标、过程、结果、差距、原因、对策、计划、验收
3. **语言要通俗、清爽、可执行**：避免空话、口号、泛泛而谈
4. **能量化尽量量化**：不能量化则用"可观察指标/现象"描述
5. **正常不展开，重点写异常/差距/关键节点**：但必须保证第三方能看懂来龙去脉
6. **每个关键结论尽量配一条证据**：原话/数据/现象/日志片段，证据不足就标【证据不足】

## 输出格式（必须严格按以下结构）

### # 复盘报告标题：{用一句话概括事件+时间范围}

### ## 0. 一句话结论（先给结论）
- 用1句话说清：这次最关键的得失是什么（不要超过30字）

### ## 1. 事件概述（让第三方立刻进入场景）
- 发生了什么：{一句话}
- 时间范围：{起止时间/阶段}
- 涉及对象/场景：{人/系统/业务/环境，尽量中性描述}
- 为什么要做这件事：{动机/触发点}

### ## 2. P｜Plan 计划（当时怎么想）

#### 2.1 目标（Goal）
- 目标1：
- 目标2：（可选）
（要求：每个目标都配"成功标准/验收点"）

#### 2.2 成功标准（验收点）
- 验收点A（可观察/可测试）：
- 验收点B：
（要求：写成"看到什么=成功"，不要写"提升、优化"这种虚词）

#### 2.3 关键假设 & 约束
- 关键假设（最多3条）：
- 关键约束（资源/权限/时间/预算/信息）：

#### 2.4 计划路径（最多5步）
- Step1：动作 → 预期输出 → 验收信号
- Step2：…
（要求：每步都有"验收信号"，哪怕很小）

### ## 3. D｜Do 执行（事实时间线）
- 用时间线列出5–15条关键节点（只写事实：做了什么→发生了什么→输出/结果/现象）
- 每条节点用标签标记：
  - ✅有效推进 / ❌无效尝试 / ⚠️异常发生 / 🔁方向调整 / 🧩关键决策
- 对每个"异常节点"，补一句：当时采取的临时处理是什么（如有）

### ## 4. C｜Check 检查（结果对照 & 差距）

#### 4.1 结果总览
- 目标达成情况：{达成/部分达成/未达成}
- 用"验收点对照表"写清楚：
  - 验收点A：现状=？ → 结论=通过/不通过 → 证据=
  - 验收点B：…

#### 4.2 差距清单（Gap List）
- Gap1：原本期望是什么 → 现实是什么 → 差距表现（可观察）→ 影响
- Gap2：…

#### 4.3 原因分析（只选一种方法深入即可）
- 请选择最适合本事件的1种方法来写（5Why / 假设驱动 / 鱼骨图思路 / A3 / DMAIC等）
- 输出必须包含：
  - 表层原因（直接触发）：
  - 机制原因（流程/信息/协作/系统/能力层）：
  - 根因（如果不解决，下次还会复发的那一层）：
- 每层原因尽量配证据；无证据标【证据不足】

#### 4.4 做得好的（可复用）
- 亮点1（做法 + 为什么有效 + 可复用条件）
- 亮点2（至少2条）

### ## 5. A｜Act 改进行动（把复盘变成下一次更强）

#### 5.1 本轮结论导向的"修正目标"
- 下一轮最关键要修的是什么（1–2条，必须可验收）

#### 5.2 行动方案（Plan of Action）
- 方案原则：{1–3条}
- 具体动作清单（写成checklist，10条以内）：
  - [ ] 动作1（负责人/截止/验收信号）
  - [ ] 动作2…

#### 5.3 风险与预案
- 风险1：触发条件 → 可能后果 → 预警信号 → 应对预案
- 风险2：…

#### 5.4 验收与回归检查
- 最终验收标准（必须具体可测）：
- 回归检查清单（防止复发，5条以内）：

#### 5.5 沉淀资产（复用输出）
- 这次沉淀为：SOP / 清单 / 模板 / FAQ / 判断规则（列出名称 + 适用场景）
- 一句话复用法：下次遇到类似事件，先做哪3件事？

### ## 6. 待确认信息（如果素材不足）
- 待确认1：需要什么信息？为什么需要？
- 待确认2：…
（要求：不提问也可以，但要把"缺口"列清楚）

## 与其他Skill的关系
- **复盘助手**：复盘助手侧重分析过程，本Skill侧重输出正式文档
- 可串联使用：先用复盘助手深度分析，再用本Skill生成正式报告
- 本Skill输出可直接交付第三方，格式规范、可读性强

## 示例片段

### 一句话结论
项目延期2周，根因是需求管理流程缺失，已建立变更控制机制。

### 验收点对照表
| 验收点 | 现状 | 结论 | 证据 |
|--------|------|------|------|
| 核心功能上线 | 已完成 | 通过 | 生产环境截图 2024-01-15 |
| 性能达标 | P99=1.2s | 不通过 | 监控平台数据，目标1s |

### 时间线示例
- 2024-01-01 ✅ 项目启动，确定排期
- 2024-01-05 ⚠️ 需求变更1：新增支付模块
- 2024-01-10 🔁 方向调整：砍掉非核心功能
- 2024-01-15 ❌ 性能测试未通过，延期

### 沉淀资产示例
- SOP：《需求变更评估流程》- 适用于所有项目
- 清单：《上线前检查清单》- 适用于版本发布
- 一句话复用法：下次先做 1) 需求冻结签字 2) 变更影响评估 3) 排期重新确认
