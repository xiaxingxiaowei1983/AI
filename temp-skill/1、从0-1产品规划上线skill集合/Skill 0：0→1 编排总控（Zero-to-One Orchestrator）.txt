# Skill 0：0→1 编排总控（Zero-to-One Orchestrator）

## 它是什么（大白话）
这是“从一个想法到能上线”的总导航/总指挥。  
它不会自动写代码，也不会自动上线；它负责：
- 告诉你下一步该启动哪个 skill（按顺序）
- 每一步必须产出什么“交付物”（文档/清单/验收点）
- 任一步没达标就先停，避免你越做越乱、最后一起爆炸

## 何时调用（触发条件）
- 你刚有一个新想法，想做成产品（例如“盲眼小郎中”）
- 你想把0→1流程标准化，减少返工与token消耗
- 你想明确：什么时候该继续、什么时候该停止/转向

## 输入（固定表单）
- 产品名：
- 目标形态：小程序 / H5 / 两者
- 目标用户（一句话）：
- 目标场景（一句话）：
- 你最担心的风险（选1个）：没人用 / 技术做不出来 / 审核过不了 / 上线不稳 / 其他
- 你期望的“上线成功标准”（一句话）：例如“用户能完成一次语音问诊闭环”

## 编排规则（硬规则）
- OR1：只按顺序走：Skill 1 → Skill 2 → Skill 3 → Skill 4
- OR2：任何一步交付物不完整：停止进入下一步
- OR3：在 Skill 4 之前，不做大规模开发（避免没验证就开工）
- OR4：完成 Skill 4 后，才进入“工程落地”，并复用你们现有技能包：
  - 套壳与隔离：Shell Reuse Orchestrator + 套壳 Skill 1–6
  - 发布与上线：H5 上线 Skill 1–7 / 小程序上线 Skill 1–7

## 执行流程（固定顺序）
### Step 1：启动 Skill 1（Idea Freeze Spec：一句话定位 & 边界）
- 你要产出的交付物：
  - 一句话定位（给谁/解决什么/怎么解决）
  - 不做清单（至少3条，防止功能膨胀）
  - 上线成功标准（可验证）
- 通过标准：
  - 三项都写清楚，不含“模糊词”（例如“做得更好”）

### Step 2：启动 Skill 2（Validation Kit：用户验证脚本）
- 你要产出的交付物：
  - 访谈/调研问题清单（10题以内）
  - 你要验证的3个假设（例如“用户愿意每天用/愿意授权录音/愿意付费”）
  - 继续/停止/转向的判定规则（写死）
- 通过标准：
  - 你能用它去问真实的人，并拿到“证据”，而不是自嗨

### Step 3：启动 Skill 3（PRD & MVP Cutter：PRD→MVP切割）
- 你要产出的交付物：
  - MVP 功能清单（只保留闭环）
  - 非 MVP 清单（明确不做）
  - MVP 验收用例（5–10条）
- 通过标准：
  - MVP 能让用户完成一次完整闭环（否则继续砍）

### Step 4：启动 Skill 4（Tech Freeze Spec：技术方案冻结）
- 你要产出的交付物：
  - 形态与入口冻结（域名/API/WS/协议/路径）
  - 数据格式冻结（尤其音频/实时）
  - 权限与隐私合规口径（小程序）
  - 后端模块边界与依赖清单（DB/存储/第三方）
- 通过标准：
  - 入口与格式不再“口头改”，变更必须升版本

## Step 5（复用，不重新造）：工程落地技能包
> 这一段不是新 skill，是复用你们已经生成好的集合。

- 套壳与隔离：Shell Reuse Orchestrator + 套壳 Skill 1–6
- 发布与上线：
  - 目标形态是 H5：用 H5 上线 Skill 1–7
  - 目标形态是小程序：用小程序上线 Skill 1–7

## 输出（固定：0→1进度记录单）
- 0→1 Record：v1
- 产品名/形态：
- Step1 定位冻结：PASS/FAIL（证据：一句话定位+不做清单+成功标准）
- Step2 用户验证：PASS/FAIL（证据：访谈记录/结论+判定规则）
- Step3 MVP切割：PASS/FAIL（证据：MVP清单+验收用例）
- Step4 技术冻结：PASS/FAIL（证据：入口/格式/合规/依赖）
- 结论：进入工程落地 / 暂停或转向（原因一句话）

## 你怎么“启动它”（一句话口令）
- “启动 Skill 0：0→1 编排总控。我要做产品【___】，形态【小程序/H5】。请按 Step1→Step4 带我走，每步给我填空模板与通过标准。”
