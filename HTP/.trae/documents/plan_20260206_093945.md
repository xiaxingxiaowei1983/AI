## 问题分析

用户遇到了 `413 Payload Too Large` 错误，这是因为：

1. **前端配置问题**：
   - `.env` 文件中 `VITE_BACKEND_BASE_URL` 设置为 `/api/htp`
   - 前端请求被发送到 `http://localhost:5174/api/htp/analyze`（前端开发服务器）
   - 前端开发服务器没有配置代理，无法处理该路径
   - 即使能处理，前端服务器的请求体大小限制可能较小

2. **后端配置**：
   - 后端服务器运行在 `http://localhost:3000`
   - 后端已正确设置 `express.json({ limit: '20mb' })`
   - 但前端请求没有正确路由到后端

## 解决方案

### 方案1：修改前端配置（推荐）
- **修改 `.env` 文件**：将 `VITE_BACKEND_BASE_URL` 更新为后端服务器的完整地址
  ```
  VITE_BACKEND_BASE_URL=http://localhost:3000/api/htp
  ```
- **优点**：配置简单，直接生效
- **缺点**：需要在不同环境中修改配置

### 方案2：配置前端代理
- **修改 `vite.config.ts` 文件**：添加代理配置
  ```typescript
  export default defineConfig({
    // 其他配置...
    server: {
      proxy: {
        '/api/htp': {
          target: 'http://localhost:3000',
          changeOrigin: true
        }
      }
    }
  });
  ```
- **优点**：前端可以继续使用相对路径，无需修改 `.env`
- **缺点**：需要修改 Vite 配置

### 方案3：优化图片处理
- **在前端添加图片压缩逻辑**：使用 Canvas API 对图片进行压缩
  - 降低图片分辨率
  - 调整图片质量
  - 减少 base64 数据大小
- **优点**：减少网络传输量，提高响应速度
- **缺点**：增加前端代码复杂度

## 推荐实施步骤

1. **优先选择方案1**：修改 `.env` 文件，直接指向后端服务器
2. **验证修复**：重启前端开发服务器，测试上传功能
3. **如果需要进一步优化**：考虑方案3，添加图片压缩逻辑

这样可以确保前端请求正确路由到后端服务器，充分利用后端的 20MB 大小限制，解决 413 错误问题。
