# Skill 6：编排 Skill（Release Orchestrator）

## 它是什么
这是一个“总控编排 Skill”，不负责解决某一个具体技术点，而是负责：
- 按顺序调用 Skill 1–5
- 每一步都产出固定格式的记录（证据链）
- 任一步 FAIL 就立刻停止，避免进入“长对话试错”
- 最终产出一份完整的《发布记录单》，方便复盘与回滚

## 它叫什么（编号建议）
- 推荐命名：**Skill 6：编排 Skill（Release Orchestrator）**
- 不建议叫 Skill 0：因为它不是一个独立前置校验，而是“串联器”
- 使用方式：发布开始时先用它“开局”，然后按它要求依次执行 Skill 1–5

## 它在流程里是什么位置
- **“调用顺序上”：它通常是第一个被调用**
- **“执行内容上”：它会依次要求你完成 Skill 1 → 2 → 3 → 4 → 5**
- 只要任意一步 FAIL：本次发布立即停止（可选择回滚）

## 何时调用（触发条件）
- 每一次准备上线（staging/prod）
- 每一次更换域名/证书/反代路径/端口/实时路径之后
- 你想要“稳定、BUG 少、返工少、token 少”的发布节奏时

## 输入（固定表单）
- 项目名：
- 发布类型：`前端发版` / `后端发版` / `全量发版`
- 目标环境：`staging` / `prod`
- 计划发布时间点（可选）：`____`
- 负责人：
- Freeze Spec 版本号（若已有）：v__
- 是否启用实时：是/否
- 是否涉及音频：是/否
- 是否启用 CDN：是/否

## 编排规则（硬规则）
- OR1：必须按顺序执行：1 → 2 → 3 → 4 → 5（不允许跳步上线）
- OR2：任一步 FAIL：立刻停止，不进入下一步
- OR3：每一步必须有“证据”，没有证据视为 FAIL
- OR4：最终必须产出《发布记录单》（固定格式）
- OR5：如果是 prod：Smoke Test 必须包含“无痕/新设备验证”（防缓存）

## 执行流程（按顺序）
### Step 1：执行 Skill 1（上线入口冻结 / Freeze Spec）
- 目标：冻结域名/协议/API 入口/实时路径/端口
- 输出物：Freeze Spec 结果单 v__
- Step 1 通过标准：
  - 入口字段填写完整
  - 规则≥6条，验收点≥6条
- 若 FAIL：停止（原因通常是入口不清晰/口径不统一）

### Step 2：执行 Skill 2（发布前置校验门 / Preflight Gate）
- 目标：残留/环境变量/Nginx/端口/实时可达的一票否决
- 输出物：Preflight 报告单（PASS/FAIL + 证据）
- 若 FAIL：停止，按 Preflight 的前三定位路径处理后重跑 Step 2

### Step 3：执行 Skill 3（反代模板校验 / Nginx Template Check）
- 目标：静态托管、SPA 刷新、/api、实时 Upgrade、HTTPS、缓存策略
- 输出物：Nginx Check 报告单
- 若 FAIL：停止，先修复 Nginx/路径/证书/缓存策略，再重跑 Step 3

### Step 4：执行 Skill 4（后端上线运行契约 / Server Runtime Contract）
- 目标：端口固定、env 权威来源、守护方式、日志、健康检查、重启生效证据
- 输出物：Runtime Contract 结果单 v__
- 若 FAIL：停止，先解决“读哪份 env/如何重启/如何确认生效”

### Step 5：执行 Skill 5（上线冒烟验收 / Smoke Test）
- 目标：8 条固定用例验收上线是否成功（页面/API/实时/音频/缓存）
- 输出物：Smoke Test 报告单
- 若 FAIL：停止，进入“回滚/定位”，不对外扩量

## 输出（固定：发布记录单）
> 这是本编排 skill 的最终交付物，你每次发布都保存一份。

- 发布记录单：v1
- 项目名 / 环境：
- 发布类型（前端/后端/全量）：
- 发布版本标识（git hash/版本号）：
- Freeze Spec：PASS/FAIL（版本 v__，证据：___）
- Preflight Gate：PASS/FAIL（证据：___）
- Nginx Template Check：PASS/FAIL（证据：___）
- Server Runtime Contract：PASS/FAIL（版本 v__，证据：___）
- Smoke Test：PASS/FAIL（证据：___）
- 结论：允许上线 / 阻断发布
- 若阻断：前三定位路径汇总：P1___ / P2___ / P3___
- 回滚策略（若需要）：回滚到版本 ___，执行人 ___，回滚验收 ___

## 验收点（本编排 skill 是否成功）
- A1：产出完整发布记录单（每步都有证据）
- A2：任一步失败都能明确停在哪一步、为什么失败、下一步看哪里
- A3：PASS 时你能有信心对外开放；FAIL 时你能快速回滚/定位

## 使用示例（最省心的用法）
- 每次发布开始：先调用本 skill，生成“发布记录单骨架”
- 然后照顺序把 Skill 1–5 的输出逐条填回记录单
- 记录单保存到：发布群/文档/仓库 release notes（任选一个固定位置）
