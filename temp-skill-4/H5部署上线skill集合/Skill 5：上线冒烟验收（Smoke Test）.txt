# Skill 5：上线冒烟验收（Smoke Test）

## 目标
用最少的用例，在最短时间内回答一个问题：**这次上线到底成功了吗？**
- 成功：可以对外开放/扩量
- 失败：立刻阻断，进入定位/回滚（不靠“感觉差不多”）

> 你们这次反复返工的一个重要原因，就是缺少固定的“上线后必跑验收”。

## 何时调用（触发条件）
- 每次部署完成后（前端/后端/反代任一变更都算一次发布）
- 改域名/证书/HTTPS/反代路径/后端端口/WS 路径之后
- 出现这些现象时：
  - “有人能用，有人不能用”
  - “页面能开但功能不工作”
  - “接口通了但实时不通”
  - “我明明上线了，但还是旧版本”

## 输入（固定表单）
- 目标环境：`staging` / `prod`
- H5 站点 URL（含协议）：`https://h5.xxx.com`
- API 策略：`/api` 同域反代 / `独立API域名`
- API Base（若独立域名）：`https://api.xxx.com`
- 轻量接口路径（若有）：`/api/health`（没有就填“任一轻量接口路径”）
- 关键业务接口（写 3 条）：`接口1/接口2/接口3`
- 是否启用实时：`是/否`
- 实时类型：`Socket.IO` / `原生WS`
- 实时路径：`/socket.io/` 或 `/ws` <mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 是否涉及音频：`是/否`
- 发布版本标识（建议填 git hash 或版本号）：`____`
- 是否启用 CDN：`是/否`

## 前置约束（硬规则）
- 任一用例失败：本次 Smoke Test 判定 FAIL
- FAIL 不继续“扩大验证范围”，直接进入“失败定位路径（前三优先）”
- 验收必须可复现：使用 **无痕窗口** 或 **全新设备** 至少跑一遍（防缓存）

## 用例清单（建议固定 8 条；按顺序执行）
> 每条都必须输出：✅/❌ + 证据（截图/响应码/日志关键字/Network 记录）

### Case 1：H5 首页可访问
- 操作：打开 `H5 URL`
- 通过标准：
  - 页面加载完成
  - Network 无关键资源 404（js/css）
- 证据：Network 截图或“404=0”的说明

失败时前三定位路径：
1) Nginx 静态目录是否部署正确
2) CDN/缓存是否未刷新（若启用）
3) 构建产物是否缺失/路径错误

### Case 2：SPA 路由刷新不 404（如是 history 路由）
- 操作：进入一个二级路由页面，按刷新
- 通过标准：仍返回页面（不出现 404/白屏）
- 证据：刷新后状态 + Network/返回码

失败时前三定位路径：
1) Nginx 是否缺少 history 回退（try_files）
2) 是否有反向代理规则截断路由
3) 前端路由模式是否与配置不一致

### Case 3：Mixed Content 为 0（HTTPS 相关硬指标）
- 操作：打开浏览器 Console
- 通过标准：Mixed Content = 0 条；无“被阻止的不安全请求”
- 证据：Console 截图

失败时前三定位路径：
1) 前端是否仍引用 http 资源/API
2) 证书/跳转是否导致部分请求走回 http
3) 实时通道是否 ws 而不是 wss

### Case 4：轻量 API 返回 200
- 操作：访问轻量接口（例如 `/api/health`）
- 通过标准：返回 200（或你们定义的成功码）
- 证据：响应码 + 返回内容摘要

失败时前三定位路径：
1) Nginx `/api` 反代是否指向正确上游端口
2) 后端进程是否 running & 端口是否监听
3) 鉴权/网关是否拦截（401/403）

### Case 5：关键业务链路（最少 3 条）
- 操作：按真实用户路径完成 3 个核心动作（例如：登录→创建→查询）
- 通过标准：3 条都成功；无致命报错
- 证据：每条的成功截图/返回码/日志关键字

失败时前三定位路径：
1) 后端日志：是否报错栈/依赖连接失败
2) env 是否读对（是否还是旧配置）
3) 数据库/第三方依赖是否可达

### Case 6：实时连接可用（若启用）
- 操作：触发一次实时连接（打开页面或点“开始连接”）
- 通过标准：
  - Socket.IO：连接建立成功，能发/收一条消息
  - 原生 WS：wss 握手成功，能收一条响应/心跳
- 证据：前端日志 + 后端 connect 日志关键字 <mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>

失败时前三定位路径：
1) 路径是否一致（`/socket.io/` 或 `/ws`）
2) Nginx 是否配置 Upgrade（接口通但实时不通的核心）
3) https 页面是否导致 ws 被拦截（必须 wss）

### Case 7：音频链路（若涉及音频）
- 操作：录音→停止→上传→拿到后端响应
- 通过标准：
  - 权限弹窗正常
  - 上传成功（非 413/timeout）
  - 后端有明确“收到音频”的日志/返回结果
- 证据：一次完整成功记录（截图/返回结果/日志关键字）

失败时前三定位路径：
1) HTTPS 与权限（非 HTTPS 常见直接异常）
2) Nginx body 大小/超时（413/504/超时）
3) 音频格式与后端解析（H5 常见 ArrayBuffer）<mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>

### Case 8：版本一致性（防旧包）
- 操作：无痕窗口打开 + 换设备打开（或清缓存后打开）
- 通过标准：看到的版本标识与本次发布一致（建议页面某处展示版本号/构建时间）
- 证据：版本号截图

失败时前三定位路径：
1) CDN 未刷新（若启用）
2) `index.html` 缓存过长
3) 服务器静态目录未替换到新 dist

## 输出格式（固定：Smoke Test 报告单）
- Smoke Test：PASS / FAIL
- 环境：staging/prod
- 发布版本标识：
- Case1 首页：✅/❌（证据：___）
- Case2 刷新：✅/❌（证据：___）
- Case3 Mixed Content：✅/❌（证据：___）
- Case4 轻量 API：✅/❌（证据：___）
- Case5 关键链路：✅/❌（证据：___）
- Case6 实时：✅/❌（证据：___）
- Case7 音频：✅/❌（证据：___）
- Case8 版本一致：✅/❌（证据：___）
- 若 FAIL：本次发布阻断；优先定位路径：P1___ / P2___ / P3___

## 完成标准（本 skill 完成的判定）
- 你生成了一份 Smoke Test 报告单
- PASS：8 条用例全部通过（或你们明确允许跳过的项已标注“Not Applicable”）
- FAIL：至少一个用例失败且已输出“前三定位路径”，不进入下一轮随意试错
