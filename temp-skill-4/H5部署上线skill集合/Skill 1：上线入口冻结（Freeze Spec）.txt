# Skill 1：上线入口冻结（Freeze Spec）

## 目标
在开始部署/联调之前，先把“线上会用到的所有入口”写成一份**冻结规格**，作为后续所有人的唯一标准答案：
- 前端只按这份规格发请求
- Nginx 只按这份规格做反代
- 后端只按这份规格监听端口/暴露路径
- 验收只按这份规格判定通过/失败

> 你们这次反复返工的根因之一，就是“入口口径在变、且没有唯一标准”。

## 何时调用（触发条件）
- 第一次准备把 H5 上到生产环境（prod）
- 你要改任何“入口相关”的东西：
  - 域名、HTTPS、API 路径、Socket.IO/WS 路径、后端端口、是否走 `/api` 反代
- 出现这些现象时：
  - “本地通，线上不通”
  - “接口通了，实时不通”
  - “我改了地址但没生效/有人还是旧的”

## 输入（固定表单，一次填完）
> 不确定的项也要先填一个“暂定值”，然后通过“版本升级”改，不允许口头随改随用。

### A. 环境信息
- 项目名：
- 目标环境：`dev` / `staging` / `prod`
- 发布形态：`H5`（本 skill 以 H5 为主）

### B. 用户访问入口（H5）
- H5 站点 URL（必须含协议）：`https://____`
- 是否强制 HTTPS：`是/否`（推荐：是）
- 是否启用 CDN：`是/否`

### C. API 入口（只允许二选一，必须选）
- 方案 1：同域反代（推荐）
  - API 对外路径前缀：`/api`
  - 对外示例：`https://<H5域名>/api/...`
- 方案 2：独立 API 域名
  - API Base URL：`https://api.____`
  - 对外示例：`https://api.____/...`

### D. 后端真实运行（服务器内部）
- 后端类型：`NestJS` / `其他`
- 后端监听端口：`____`（例：3006）
- 后端监听地址：`127.0.0.1` / `0.0.0.0`
- 健康检查路径（若有）：`/health` 或 `无`
- 日志查看方式（写一个即可）：`pm2 logs` / `journalctl` / `文件路径`

### E. 实时通道（只在启用时填写）
- 是否启用实时：`是/否`
- 实时类型：`Socket.IO` / `原生 WebSocket`
- 实时路径（固定）：
  - Socket.IO：`/socket.io/`（推荐保持默认）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
  - 原生 WS：`/ws`（若你们用的是这个）<mccoremem id="03fl75psdph7oiq63zur4f5vg" /></mccoremem>
- 对外完整示例（写一个）：
  - Socket.IO：`https://<H5域名>/socket.io/`
  - 原生 WS：`wss://<H5域名>/ws`

### F. 音频/语音（如有）
- 是否涉及录音/音频上传：`是/否`
- H5 上行格式：`ArrayBuffer/TypedArray/base64/其他`（你们 H5 常见是 ArrayBuffer）<mccoremem id="01KHFQYTH1828XWQA5AAM4K3YS" /></mccoremem>

## 冻结规则（不可变规则，写死在文档里）
至少写 6 条；下面给你可直接用的模板：

- R1：H5 页面如果是 `https://`，则 API 与实时必须是安全通道（https / wss），禁止 Mixed Content。
- R2：前端只允许使用“唯一 API 入口”（选了 `/api` 就全站 `/api`；选了独立域名就全站独立域名），禁止混用。
- R3：前端代码/配置中不得出现 `localhost`、`127.0.0.1`、旧域名、旧端口（上线前必须 0 命中）。
- R4：实时路径固定（Socket.IO 固定 `/socket.io/` 或 WS 固定 `/ws`），不得临时改。
- R5：对外只开放 80/443（如采用 Nginx 反代）；后端真实端口不直接暴露公网（如适用）。
- R6：任何入口变更必须“升版本”（v1→v2），禁止口头变更导致多人口径不一致。

## 输出（固定格式：Freeze Spec 结果单）
> 你填完输入表单后，把下面这段完整输出保存起来。

- Freeze Spec 版本：v1
- 项目名 / 环境：
- H5 站点 URL：
- API 入口（`/api` 同域反代 或 独立 API 域名）：
- 后端真实端口：
- 实时类型/路径（如启用）：
- 强制 HTTPS：是/否
- 规则（R1~R6）：已确认
- 验收点（A1~A6）：已定义

## 验收点（可量化，必须能判定“过/不过”）
至少 6 条；下面给你可直接用的模板：

- A1：打开 `H5 URL` 成功；浏览器 Console 无红色致命错误。
- A2：Console 中 Mixed Content = 0 条。
- A3：访问一个轻量 API（例如 `/api/health` 或你们任意轻量接口）返回 200。
- A4：关键业务接口 3 条（写明路径/功能）全部返回成功。
- A5：若启用实时：Socket.IO/WS 连接建立成功，并完成一次收发（有明确证据：前端日志或后端 connect 日志）。
- A6：换无痕窗口/新设备访问仍能拿到最新版本（避免缓存导致“我明明上线了但还是旧的”）。

## 失败时优先定位（只看前三个，避免陷入长排查）
- P1：浏览器 Network：请求是否命中正确域名/路径；是否 301/302；是否 Mixed Content。
- P2：Nginx 日志：对应路径是否命中；实时通道是否 Upgrade（若启用）。
- P3：后端日志：是否收到请求/握手；是否端口监听；是否鉴权/跨域/内部错误。

## 完成标准（本 skill 完成的判定）
- 你能拿出一份“Freeze Spec 结果单（v1）”，并且：
  - 所有入口字段都填满（H5/API/端口/实时）
  - 规则 ≥ 6 条
  - 验收点 ≥ 6 条且可量化
- 后续任何联调/部署问题，都能先对照这份 Freeze Spec 判断“是不是入口跑偏”
